{{/* 01. what is a pax */}}
{{ partial "section-head.html" (dict "caption" "01.what-is-a-pax" "title" "What is a pax") }}

A pax — Portable Analytical eXpertise — is a structured, machine-readable knowledge package that captures the accumulated empirical knowledge of a research domain. It is not a paper, a database, or a model weight file. It is a folder of flat YAML and JSON that any agent, any script, or any human can read without tooling.

The central insight is portability. Frontier language models know surface-level facts about a lot of domains. They do not know which claims are well-identified, which findings have replicated, or what the field's canonical vocabulary even is. A pax answers all three questions: here is the vocabulary, here are the findings, here is who said what and how they measured it.

Paxes are evidence-backed. Every finding must point to a source entry with a resolvable DOI (or a declared caveat when the literature does not provide one). The linter enforces referential integrity at publish time — a finding cannot cite a source that does not exist in sources.json.

Paxes are composable. An agent running a playbook can import multiple paxes and resolve constructs across them using the shared construct registry. The registry detects near-duplicate ids and warns authors before collision enters the shared namespace.

{{/* 02. anatomy */}}
{{ partial "section-head.html" (dict "caption" "02.anatomy" "title" "Anatomy of a pax") }}

Every pax is a directory. The directory name is the pack's kebab-case id. Inside it you will always find a manifest, and usually find the five knowledge files plus a playbooks subdirectory. Files are flat JSON — no proprietary database, no encoding tricks, no binary blobs.

{{ partial "code-block.html" (dict "lang" "text" "id" "anatomy-tree" "code" "my-pack-name/\n pax.yaml # manifest (required)\n knowledge/\n domain.json # domain definition (required)\n constructs.json # construct vocabulary (required)\n findings.json # empirical claims (required if sourced)\n sources.json # paper/report metadata (required if findings)\n propositions.json # theoretical claims (optional)\n construct_relationships.json # causal/correlational links (optional)\n playbooks/\n quick_start.yaml # analysis workflow (optional)") }}

The simplest valid pax has a manifest, a domain file, and a constructs file. Add findings, sources, and a playbook when you have the evidence. The linter will tell you what is missing for each pax type at validation time.

pax types

{{/* 03. constructs */}}
{{ partial "section-head.html" (dict "caption" "03.constructs" "title" "Authoring constructs") }}

Constructs are the vocabulary of a pax. Each construct entry defines one concept or variable: what it is, how it is measured, what other names it goes by. Construct ids must be snake_case ASCII, no leading underscore, at most 48 characters. The linter checks the shared registry for near-duplicate ids and warns on collision before publish.

{{ partial "code-block.html" (dict "lang" "json · knowledge/constructs.json" "id" "constructs-ex" "code" "[\n {\n \"id\": \"gdp_per_capita\",\n \"display_name\": \"GDP Per Capita\",\n \"construct_type\": \"quantifiable\",\n \"definition\": \"Gross domestic product divided by total population, measured in constant 2015 USD PPP.\",\n \"measurement_level\": \"ratio\",\n \"aliases\": [\n { \"alias\": \"GDP/capita\", \"alias_type\": \"abbreviation\" },\n { \"alias\": \"per capita income\", \"alias_type\": \"synonym\" }\n ]\n },\n {\n \"id\": \"life_expectancy\",\n \"display_name\": \"Life Expectancy at Birth\",\n \"construct_type\": \"outcome\",\n \"definition\": \"Average number of years a newborn is expected to live under current mortality conditions.\",\n \"measurement_level\": \"ratio\",\n \"aliases\": [\n { \"alias\": \"LE\", \"alias_type\": \"abbreviation\" }\n ]\n }\n]") }}

Required fields: id, display_name, construct_type, definition.

construct_type values
typeuse when
quantifiablemeasured on a numeric scale (ratio or interval)
conceptabstract, not directly measurable
processa mechanism or causal pathway
compositean index or aggregated score
outcomea dependent variable in the field's literature
{{/* 04. findings */}}
{{ partial "section-head.html" (dict "caption" "04.findings" "title" "Authoring findings" "kicker" "One entry per empirical claim. Never aggregate two claims into one finding.") }}

Findings encode what the literature actually found. Each entry is a single, falsifiable empirical claim tied to specific constructs, a specific source, and a specific confidence level. The finding_text field is a full sentence in plain English. Do not start with "The study found that" — state the relationship directly.

{{ partial "code-block.html" (dict "lang" "json · knowledge/findings.json" "id" "findings-ex" "code" "[\n {\n \"source_id\": \"author_year_2024\",\n \"domain_id\": \"my_domain\",\n \"finding_text\": \"GDP per capita is positively associated with life expectancy (beta=0.42, SE=0.08, p<0.001, N=1200).\",\n \"construct_ids\": [\"gdp_per_capita\", \"life_expectancy\"],\n \"direction\": \"positive\",\n \"confidence\": \"strong\",\n \"method_used\": \"OLS regression with country fixed effects\",\n \"effect_size_value\": 0.42,\n \"effect_size_se\": 0.08,\n \"effect_size_type\": \"beta\",\n \"p_value\": 0.001,\n \"sample_size\": 1200,\n \"model_specification\": \"OLS with country and year FE, clustered SE\",\n \"covariates_controlled\": [\"population\", \"trade_openness\"]\n }\n]") }}

confidenceenumyesweak / moderate / strong
effect_size_valuenumbernobeta, OR, Cohen's d, etc.
effect_size_senumbernostandard error of the estimate

{{/* 05. sources */}}
{{ partial "section-head.html" (dict "caption" "05.sources" "title" "Sources & citations") }}

Every finding must have a source. Sources live in sources.json and are referenced by id from findings. The id convention is author_year — for multi-author papers use first-author only (e.g., fearon_laitin_2003). For reports or grey literature, use org_year (e.g., worldbank_2022).

{{ partial "code-block.html" (dict "lang" "json · knowledge/sources.json" "id" "sources-ex" "code" "[\n {\n \"id\": \"author_year_2024\",\n \"source_type\": \"academic_paper\",\n \"title\": \"The Full Paper Title Goes Here\",\n \"authors\": \"Last, First; Last2, First2\",\n \"year\": 2024,\n \"doi\": \"10.1234/example.doi\",\n \"methodology_summary\": \"Panel regression with country/year FE, N=1200\",\n \"sample_size\": 1200,\n \"study_design\": \"observational_longitudinal\",\n \"open_access\": true\n }\n]") }}

Required: id, title, authors, year. Strongly encouraged: doi, study_design. The linter resolves DOIs against Crossref and flags unresolvable ones as warnings. Open-access papers must set open_access: true and must be linkable at publish time. Paywalled papers must declare open_access: false — do not leave it null.

Grey literature (reports, working papers, blog posts) is permitted when it is the best available evidence, but each such entry must include a methodology_summary explaining why it was included and what its limitations are.

{{/* 06. propositions */}}
{{ partial "section-head.html" (dict "caption" "06.propositions" "title" "Propositions" "kicker" "Theoretical claims linking constructs — the field's reusable rules of thumb.") }}

Propositions are higher-order theoretical statements about how constructs relate. Unlike findings, they are not tied to a single study; they represent the accumulated theoretical consensus (or debate) in the field. Propositions link to the findings that support them via an evidence array.

{{ partial "code-block.html" (dict "lang" "json · knowledge/propositions.json" "id" "propositions-ex" "code" "[\n {\n \"id\": \"P001\",\n \"text\": \"In multi-tier networks, betweenness centrality of a node predicts its contribution to cascading failure size, conditional on tier depth.\",\n \"kind\": \"theoretical\",\n \"supports\": [\"cascading_failure_size\"],\n \"evidence\": [\"F003\", \"F011\", \"F042\"]\n },\n {\n \"id\": \"P002\",\n \"text\": \"Geographic concentration of refining (not extraction) is the dominant driver of strategic-mineral supply risk.\",\n \"kind\": \"empirical\",\n \"evidence\": [\"F007\", \"F022\"]\n }\n]") }}
proposition kinds
kinduse when
theoreticalthe claim is a formal or informal theory not yet fully tested
empiricalthe claim is directly supported by one or more findings
methodologicalthe claim is about how to measure or analyze the constructs
{{/* 07. playbooks */}}
{{ partial "section-head.html" (dict "caption" "07.playbooks" "title" "Playbooks · YAML DSL" "kicker" "Step-by-step recipes the agent runs end-to-end.") }}

A playbook is a reproducible analysis workflow defined in YAML. It specifies inputs, which engines to use, and the order of steps. Each step references an engine by id and passes typed arguments. Engines are separate paxes of type "engine" — the playbook does not bundle the code, it declares the dependency.

{{ partial "code-block.html" (dict "lang" "yaml · playbooks/quick_start.yaml" "id" "playbook-ex" "code" "name: cascading-failure\nversion: 1.0\ndescription: |\n Compute centrality, identify critical nodes,\n test percolation under random and targeted failure.\ninputs:\n - id: network\n schema: edgelist\n - id: id_col\n type: string\n default: \"node_id\"\nengines:\n - network_centrality\n - percolation\nsteps:\n - id: 01_centrality\n engine: network_centrality\n args:\n measures: [betweenness, eigenvector, harmonic]\n out: centrality_table\n - id: 02_percolation\n engine: percolation\n args:\n mode: random\n thresholds: [0.05, 0.10, 0.20]\n out: failure_curve\n - id: 03_targeted\n engine: percolation\n args:\n mode: targeted\n order_by: betweenness\n out: critical_nodes\n - id: 04_report\n engine: yaml_report\n in: [centrality_table, failure_curve, critical_nodes]\n out: results.yaml\non_error: log_and_continue\noutput:\n format: yaml\n attach_findings: [F003, F011]") }}

Playbook ids follow snake_case. The quick_start.yaml name is a convention — the registry surfaces it as the primary playbook on the pack's detail page. A pack of type "engine" carries no constructs or findings, only playbooks and the engine definition itself.

{{/* 08. validation */}}
{{ partial "section-head.html" (dict "caption" "08.validation" "title" "Validation") }}

Run pax validate before submitting. The linter checks every structural constraint. It will not block on warnings, but it will block on errors. Fix all errors; review all warnings before deciding to ignore them.

{{ partial "code-block.html" (dict "lang" "bash" "id" "validate-ex" "code" "$ pax validate\n\nv pax.yaml schema ok · type=field · version=1.0.9\nv constructs.json 38 entries · all ids snake_case · no registry collisions\nv findings.json 113 entries · all source_ids resolved · all construct_ids resolved\nv sources.json 47 entries · 46/47 doi resolved\nW findings.json F042: confidence='strong' but design=observational_cross_sectional\n (suggest 'moderate'? — non-blocking)\nv propositions.json 12 entries · all evidence ids resolved\nv playbooks/ 8 files · all engines registered\n\nready 1 warning") }}

{{/* 09. peer review */}}
{{ partial "section-head.html" (dict "caption" "09.peer-review" "title" "Peer review" "kicker" "Light, machine-checked, 48–72h turnaround.") }}

The registry is open — anyone can author and submit. Review is structured but not gatekeeping. The goal is to ensure the linter passes and to catch obvious errors (miscitations, sign flips, impossible statistics) that automation misses. Reviewers do not re-run your analyses.

  1. Schema lint. Automated, runs on PR open. Required keys, type checks, source resolution, construct id collisions. Must pass before human review begins.
  2. DOI verification. DOIs resolved against Crossref. Open-access papers must be linkable; paywalled papers must declare so. Non-resolvable DOIs are flagged for author response.
  3. Editorial pass. One human reviewer skims for obvious miscitations, sign flips, or unsupported confidence labels. They do not re-do your peer review — they check you are representing the literature faithfully.
  4. Trust score. Each pax receives a public trust score (0–5) based on author credentials, source quality, replication evidence, and schema completeness. Visible on the marketplace detail page.

Estimated time from PR open to publish: 48–72h for well-formed paxes. Schema errors will extend this; fix them before opening the PR.

{{/* 10. versioning */}}
{{ partial "section-head.html" (dict "caption" "10.versioning" "title" "Versioning & deprecations") }}

Paxes use semantic versioning. The version in pax.yaml is the pack version — it is distinct from schema_version, which tracks the pax format specification. Bump pack version according to the rules below whenever you publish a change.

bumpwhen
majorremoving or renaming a construct id; removing a published finding
minoradding new constructs or findings; splitting a construct (with alias for old id)
patchtypo fixes, source-id corrections, additional aliases, metadata updates

Deprecations must live in pax.yaml for at least two minor versions before an id is removed. The deprecated field accepts an array of objects with id, deprecated_in, and reason. Any downstream pax depending on a deprecated id will receive a warning at their next validate run.

{{ partial "code-block.html" (dict "lang" "yaml · pax.yaml — deprecation block" "id" "deprecation-ex" "code" "deprecated:\n - id: old_construct_name\n deprecated_in: \"1.3.0\"\n reason: \"Renamed to new_construct_name for registry alignment.\"\n replaced_by: new_construct_name") }}
{{/* 11. publishing */}}
{{ partial "section-head.html" (dict "caption" "11.publishing" "title" "Publishing") }}

There are two paths to publishing a pax: the web uploader at submit.pax-market.com, and a direct pull request on the pax-market registry repository. Both paths run the same CI pipeline — schema lint, DOI check, construct collision check — before merging.

path 1 — web upload
  1. Zip your pack directory: zip -r my-pack.zip my-pack-name/
  2. Upload at submit.pax-market.com — sign in with GitHub.
  3. Validation runs automatically. Errors appear inline; fix and re-upload.
  4. A pull request is opened on the registry on your behalf.
  5. After editorial review, your pack is merged and live within minutes.
path 2 — direct PR
{{ partial "code-block.html" (dict "lang" "bash" "id" "publish-ex" "code" "# fork github.com/JELambert/pax-market, then:\ngit clone https://github.com/YOUR_HANDLE/pax-market\ncp -r my-pack-name pax-market/packs/\ncd pax-market\ngit checkout -b add-my-pack-name\ngit add packs/my-pack-name/\ngit commit -m \"add: my-pack-name v1.0.0\"\ngit push origin add-my-pack-name\n# open PR — CI runs automatically") }}

CI validates the pack, generates a registry release, and deploys the updated /registry.json to the CDN. Your pack's detail page appears at pax-market.com/pax/my-pack-name immediately after deploy completes.
